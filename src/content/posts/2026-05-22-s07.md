# s07 任务系统

今天在 s06 上下文压缩的基础上，搞清楚了如何把内存里的扁平清单升级成磁盘上的任务依赖图（DAG），让 Agent 的"记忆"能跨越对话存活。

---

## 核心概念

**任务图（Task DAG）**

- **直觉/类比**：就像施工项目的甘特图——地基没打完，不能建墙；两面墙可以同时砌，但封顶要等所有墙都好。每个任务是一块砖，`blockedBy` 是"我压在哪块砖上面"这条关系。
- **解决什么问题**：没有它，Agent 只知道"有哪些事"，不知道"现在能做哪件事"——要么乱序执行，要么每次都要人工说明顺序。
- **和已有知识的关系**：是对 s03 TodoManager 的扩展——从"做完没做完"的布尔值，升级为带方向边的状态机。

**依赖解除（`_clear_dependency`）**

- **直觉/类比**：像工厂流水线上的"放行信号"——上游工位完工后，自动把下游工位的"等待"标签撕掉。不需要下游主动去问，上游完成时就广播一次。
- **解决什么问题**：如果手动维护"谁解锁了谁"，稍微复杂一点就会漏更新，任务永远卡在 `blocked`。

---

## 结构图

```
                 +----------+
            +--> | task 2   | --+
            |    | pending  |   |
+----------+     +----------+    +--> +----------+
| task 1   |                          | task 4   |
| completed| --> +----------+    +--> | blocked  |
+----------+     | task 3   | --+     +----------+
                 | pending  |
                 +----------+

状态机：pending -> in_progress -> completed
```

这张图表达的核心关系：**完成是单向传播的**。task 1 完成后，同时解锁 task 2 和 task 3（并行）；只有两者都完成，task 4 的 `blockedBy` 列表才变空，才真正可以开始。"现在能做什么"这个问题，在任意时刻都能从图里直接读出来。

## 代码

### 1. `TaskManager.__init__` + `create`：任务的出生

**输入**：`tasks_dir`（存放所有任务 JSON 的目录路径）；`create` 接收任务标题 **输出**：在磁盘上创建一个 `task_N.json` 文件，返回任务的 JSON 字符串
每个任务一个 JSON 文件, CRUD + 依赖图 

CRUD： **C**reate — 创建任务（新建一个 JSON 文件）  **R**ead — 读取任务（查看任务详情）  **U**pdate — 更新任务（修改状态、标题、描述等）  **D**elete — 删除任务（删除对应 JSON 文件）

依赖图（Dependency Graph）
任务之间存在**前置关系**，用图（Graph）来表达：

```
task-001 → task-003 → task-005
             ↑
task-002 ──→ task-004
```

```python
class TaskManager:
    def __init__(self, tasks_dir: Path):
        self.dir = tasks_dir
        self.dir.mkdir(exist_ok=True)
        self._next_id = self._max_id() + 1  # 最关键一行：从磁盘已有文件推断下一个 ID，重启后不会从 1 重来

    def create(self, subject, description=""):
        task = {
            "id": self._next_id,
            "subject": subject,
            "status": "pending",   # 初始状态：待办
            "blockedBy": [],       # 初始无前置依赖
            "owner": ""            # 预留给多 Agent 认领（s09 会用到）
        }
        self._save(task)
        self._next_id += 1
        return json.dumps(task, indent=2)
```

---

### 2. `_clear_dependency`：完成时的广播解锁

**输入**：刚完成的任务 ID 
**输出**：**当一个任务完成后，把它从所有"依赖它"的任务的阻塞列表移除**。

```python
def _clear_dependency(self, completed_id):
    # 扫描目录下所有任务文件
    for f in self.dir.glob("task_*.json"):
    
        task = json.loads(f.read_text())
        # 拆开来看： text = f.read_text() # 第一步：读取 → 从磁盘拿到一个字符串 task = json.loads(text) # 第二步：解析 → 把字符串转成 Python 字典
        
        # 检查这个任务是否被 completed_id 阻塞
        if completed_id in task.get("blockedBy", []):
            
            # 从阻塞列表中移除已完成的任务ID
            task["blockedBy"].remove(completed_id)
            
            # 立刻写回磁盘（改一个存一个）
            self._save(task)
```

**最关键的一行**：`self._save(task)` 在循环内部逐个写盘，不等全部扫描完再统一保存——部分失败时，已解锁的任务不受影响。

---

### 3. **状态变更 + 依赖关联**: `update` 处理状态转换和依赖边。

**输入**：任务 ID + 新状态（可选）+ 要增减的依赖边（可选） 
**输出**：更新后写盘；若状态变为 `completed`，顺带触发依赖广播

```python
def update(self, task_id, status=None, add_blocked_by=None, remove_blocked_by=None):
    task = self._load(task_id)
    if status:
        task["status"] = status
        if status == "completed":
            self._clear_dependency(task_id)  # 最关键一行：完成即自动广播，调用方不需要手动解锁下游
    if add_blocked_by:
        task["blockedBy"] = list(set(task["blockedBy"] + add_blocked_by))  # set 去重，防止重复加边
    if remove_blocked_by:
        task["blockedBy"] = [x for x in task["blockedBy"] if x not in remove_blocked_by]
    self._save(task)
```

**最关键的一行**：`self._clear_dependency(task_id)` 藏在状态判断里——完成任务和解锁下游是一个原子动作，不会出现"完成了但忘记解锁"的情况。

### 4. `TOOL_HANDLERS`：把任务图接入 Agent 的工具分发表

**输入**：Agent 调用工具时传入的关键字参数 **输出**：路由到对应的 `TaskManager` 方法，返回结果字符串

```python
TOOL_HANDLERS = {
    # ...已有的基础工具...
    "task_create": lambda **kw: TASKS.create(kw["subject"]),          # 新建任务
    "task_update": lambda **kw: TASKS.update(kw["task_id"], kw.get("status")),  # 最关键一行：Agent 通过这里改状态、加/删依赖
    "task_list":   lambda **kw: TASKS.list_all(),                     # 列出全部任务（含依赖关系）
    "task_get":    lambda **kw: TASKS.get(kw["task_id"]),             # 查单个任务详情
}
```

**最关键的一行**：`"task_update"` 这一条——它是 Agent 推进任务图的唯一出口，状态流转、依赖增减全走这里，其余三个工具都是只读查询。

## 与 s06 的关键变化

|维度|s06 之前|s07 之后|
|---|---|---|
|存储位置|内存，压缩后丢失|磁盘 JSON，重启存活|
|任务关系|无|`blockedBy` 有向边|
|状态粒度|做完/没做完|`pending → in_progress → completed`|
|"现在能做什么"|需要人工判断|图结构直接回答|
|工具数量|5|8（新增 `task_create/update/list/get`）|

## 本章总结

> s07 将 s03 的内存扁平清单升级为磁盘持久化的任务图（DAG）。每个任务一个 JSON，通过 `blockedBy` 声明前置依赖，状态按 `pending → in_progress → completed` 流转；完成时自动清除其他任务的依赖，解锁后续任务。提供 create / update / list / get 四个工具，是后续多 agent 协作的协调骨架。
