## 模块2：概念精讲

### 概念1：张量（Tensor）

- **一句话定义**：张量就是一个"有维度的数字容器"，可以理解为带形状信息的多维数组，支持 GPU 加速和自动求导。
- **为什么要用它**：Python 列表做 `+`、`*` 是拼接和重复，不是数学运算。神经网络需要大量的矩阵乘法、加法、广播，列表根本干不了这些事。
- **与已有知识的关系**：可以理解为之前手写 NN 时用的那些矩阵（如权重矩阵 W），只不过 PyTorch 把它封装成了 `torch.Tensor` 对象，自动管理形状、设备（CPU/GPU）、梯度等。
- **类比/直觉**：列表像一排抽屉，只能存东西、取东西；张量像一个 Excel 表格，不仅能存，还能对整行整列做数学运算，而且知道"自己是几行几列"。

```python
a = [1, 2, 3]
b = torch.tensor([1, 2, 3])

a + [2]          # → [1, 2, 3, 2]  拼接！
b + torch.tensor(2)  # → tensor([3, 4, 5])  逐元素加法！

a * 2            # → [1, 2, 3, 1, 2, 3]  重复！
b * 2            # → tensor([2, 4, 6])  逐元素乘法！
```

**踩坑记录**：初学时容易把 `a * 2` 的"重复"和 `b * 2` 的"乘法"搞混。记住一句话：**列表是数据结构，张量是数学对象**，底层逻辑完全不同。

---

### 概念2：张量的形状（Shape）与维度（Dimension）

- **一句话定义**：`shape` 描述张量在每个维度上有多少个元素，从外到里数方括号的层数就是维度数。
- **为什么要用它**：形状错误是 PyTorch 最常见的 bug 来源。矩阵乘法要求维度对齐，广播要求形状兼容，不搞清楚 shape 就会报各种错。
- **类比/直觉**：
  - `shape=()` → 一个数字（标量），0 维
  - `shape=(4,)` → 一排数字（向量），1 维，**一个方括号**
  - `shape=(1,4)` → 一张只有 1 行的表（矩阵），2 维，**两个方括号**
  - `shape=(2,3,4)` → 两张表叠在一起（3 维张量）

```python
torch.tensor(5).shape       # torch.Size([])       标量
torch.tensor([1,2,3]).shape # torch.Size([3])       向量
torch.randn(2,3,4).shape    # torch.Size([2,3,4])   3维张量
```

**易混淆点**：`torch.randn(4)` 和 `torch.randn(1,4)` 看起来差不多，但前者是 1 维向量（`shape=(4,)`），后者是 2 维矩阵（`shape=(1,4)`）。在矩阵乘法 `@` 中，维度不同会导致结果不同，这个坑后面会反复踩。

---

### 概念3：索引与切片（Indexing & Slicing）

- **一句话定义**：用方括号 `[ ]` 从张量中取出子集，维度从外到里依次用 `,` 分隔。
- **与已有知识的关系**：和 Python 列表的索引逻辑一样，但扩展到了多维。

```python
d = torch.randn(2, 3, 4)  # 2张表，每张3行4列
d[:, 0, :]   # 取两张表的第1行 → shape (2, 4)
d[:, 2, :]   # 取两张表的第3行 → shape (2, 4)
```

**直觉**：`:` 表示"这个维度全要"，数字表示"只取这个位置"。可以类比为：你在 Excel 里有两张工作表，`d[:, 0, :]` 就是"两张表我都只看第 0 行"。

---

### 概念4：reshape 与 view

- **一句话定义**：改变张量的形状，但不改变其中的数据。`reshape` 更通用，`view` 要求内存连续。
- **为什么要用它**：神经网络中经常需要"把一维数据变成矩阵"或"把高维特征展平"。

```python
x = torch.arange(12)        # tensor([0,1,2,...,11])
y = x.reshape(2, 6)         # 2行6列
y = x.view(2, 2, 3)         # 2张表，每张2行3列
```

**类比**：就像把 12 块积木重新排列——先是摆成一排，然后摆成 2 行 6 列，再摆成 2 层每层 2×3。积木本身没变，只是排列方式变了。

---

### 概念5：广播（Broadcasting）

- **一句话定义**：当两个张量形状不同时，PyTorch 会自动"扩展"较小的那个，使它们能进行逐元素运算。
- **广播规则（从右往左看）**：
  1. 从最右边的维度开始逐一对比
  2. 两个维度**相等**，或者**其中一个为 1**，就可以广播
  3. 为 1 的那个维度会被"拉伸"到和另一个一样大

```python
a = torch.tensor([[1,2,3],    # shape: (2, 3)
                  [4,5,6]])
b = torch.tensor([10,20,30])   # shape: (3,)

a + b  # b 从 (3,) 广播成 (2,3)
# tensor([[11, 22, 33],
#         [14, 25, 36]])
```

**易混淆点**：`torch.randn(4)` 是 `shape=(4,)` 一维张量，`torch.randn(1,4)` 是 `shape=(1,4)` 二维张量。两者都能和 `shape=(3,4)` 的张量广播相加，但行为略有不同——不过结果相同。

**进阶广播**：当张量维度数不同时，**从右往左**对齐后，缺失的维度自动视为 1。

```python
a = torch.randn(2, 1, 3)
b = torch.randn(1, 4, 1)
(a + b).shape  # → (2, 4, 3)
# a: (2,1,3) → 广播成 (2,4,3)
# b: (1,4,1) → 广播成 (2,4,3)
```

---

### 概念6：矩阵乘法 `@` 运算符

- **一句话定义**：`@` 是 PyTorch 的矩阵乘法运算符，等价于 `torch.matmul()`。
- **核心规则**：最后两个维度做矩阵乘法（内积），前面的维度做广播。

```python
x = torch.randn(1, 3)     # (1, 3)
w = torch.randn(5, 3)     # (5, 3)

out = x @ w.T              # w.T → (3, 5)，结果 (1, 5)
```

**批量矩阵乘法**：

```python
a = torch.randn(2, 3, 4)   # 2个 (3,4) 矩阵
b = torch.randn(2, 4, 5)   # 2个 (4,5) 矩阵
c = a @ b                   # (2, 3, 5) —— 前面的 2 配对
```

**直觉**：想象你有两摞文件，每摞 2 张纸。`a @ b` 就是"第 1 摞的第 i 张和第 2 摞的第 i 张相乘"，结果还是两摞，每摞的形状变了。

### 概念7：`.item()` 方法

- **一句话定义**：从只含一个元素的张量中提取 Python 数字。
- **限制**：只能用于标量张量（shape 为 `()` 或 `(1,)`），多个元素会报错。

```python
x = torch.randn(1)
x.item()  # → 1.289（一个普通 Python 浮点数）
```

**使用场景**：在循环打印预测结果时，经常需要 `.item()` 把张量转成普通数字来显示。

---

### 概念8：`zip()` 函数

- **一句话定义**：把多个可迭代对象"拉链式"配对，每次循环同时从每个对象中取出一个元素。
- **类比**：就像拉链的两排齿，每次拉一下，左边出一个，右边出一个，配成一对。

```python
name = ['Andrej', 'karpathy']
scores = [100, 95]
for x, y in zip(name, scores):
    print(x, 'gets', y)
# Andrej gets 100
# karpathy gets 95
```

**在本项目中的用途**：`zip(Xtr[:16], Ytr[:16])` 把输入和标签配对，方便逐条查看训练数据。

### 概念9：字符级语言模型的数据集构建

- **一句话定义**：把原始文本转换成"给定前 N 个字符，预测下一个字符"的 (输入, 标签) 对。
- **为什么要用它**：神经网络不能直接吃字符串，需要把字符映射成数字。同时需要定义"上下文窗口"——模型看几个字符来预测下一个。
- **与已有知识的关系**：这是之前学的"有监督学习"的典型数据格式：X 是输入特征，Y 是标签。

**关键参数**：`block_size = 8`，表示每次用前 8 个字符作为上下文。

**数据转换过程**：

```
单词 "erim."，block_size = 8

第一步：context = [0,0,0,0,0,0,0,0]，预测 'e'(=5)
第二步：context = [0,0,0,0,0,0,0,5]，预测 'b'(=2)
第三步：context = [0,0,0,0,0,0,5,2]，预测 'r'(=18)
...
```

每个 `.` (编号 0) 既是上一个单词的结束标志，也是上下文的填充字符（padding）。

---

### 概念10：训练集 / 验证集 / 测试集划分

- **一句话定义**：把数据分成三份——训练集（80%）用来训练，验证集（10%）用来调参，测试集（10%）用来最终评估。
- **为什么要用它**：如果用同一批数据训练和评估，模型可能只是"背答案"（过拟合），在新数据上表现很差。划分数据集是为了模拟"考试时没见过的题"。

```python
n1 = int(0.8 * len(words))  # 训练集终点
n2 = int(0.9 * len(words))  # 验证集终点

Xtr, Ytr = build_dataset(words[:n1])     # 前 80%
Xdev, Ydev = build_dataset(words[n1:n2]) # 中间 10%
Xte, Yte = build_dataset(words[n2:])     # 最后 10%
```

---

### 概念11：`random.shuffle` 与 `random.seed`

- **一句话定义**：`shuffle` 打乱列表顺序；`seed` 固定随机种子，让每次运行结果一样。
- **为什么要用它**：打乱是为了消除原始数据的顺序偏差（比如数据集刚好按字母排序）。固定种子是为了**可复现**——下次运行代码，训练/验证/测试集的划分方式完全一样。

## 模块3：代码实现

### 代码段1：字符到数字的映射

**功能**：建立字符 ↔ 整数的双向映射字典，为后续的数据转换做准备。

```python
# 拿到所有出现过的字符，排序
chars = sorted(list(set(''.join(words))))

# string to integer：字符 → 数字，从1开始编号
stoi = {s: i+1 for i, s in enumerate(chars)}
stoi['.'] = 0  # 用 0 表示起始/结束符

# integer to string：数字 → 字符（反向映射）
itos = {i: s for s, i in stoi.items()}
```

**输出解读**：`stoi` 把 26 个字母映射到 1-26，`'.'` 映射到 0。`itos` 是反向字典，方便把预测的数字转回字符。

**为什么要用 0 做填充**：在上下文窗口不够长时，用 0 填充（即 `'.'`），这样模型能学会区分"真正的字符上下文"和"还没开始的填充"。

---

### 代码段2：数据集构建函数 `build_dataset`

**功能**：把一列单词转换成 (X, Y) 训练数据对。X 的每一行是一个长度为 `block_size` 的上下文窗口，Y 是该上下文对应的下一个字符。

```python
block_size = 8

def build_dataset(words):
    X, Y = [], []

    for w in words:
        context = [0] * block_size  # 初始化为8个0（填充）
        for ch in w + '.':          # 遍历单词的每个字母 + 结束符
            ix = stoi[ch]           # 把字符转成数字
            X.append(context)       # 记录当前上下文
            Y.append(ix)            # 记录要预测的目标字符
            context = context[1:] + [ix]  # 滑动窗口：去掉头，加在尾

    X = torch.tensor(X)
    Y = torch.tensor(Y)
    print(X.shape, Y.shape)
    return X, Y
```

**后置总结**：
- 训练集：`torch.Size([182580, 8])` 和 `torch.Size([182580])` → 182580 条样本，每条输入是 8 个字符编号，标签是 1 个字符编号
- 验证集：22767 条
- 测试集：22799 条

**踩坑记录**：
- `context = context[1:] + [ix]` 这行容易误解。`context[1:]` 是切片（去掉第一个元素），`+ [ix]` 是列表拼接（不是数学加法）。这里用的是**列表的操作**，不是张量的。
- 如果把 `block_size` 从 8 改小（比如 3），X 的每行就只有 3 个字符。这个值决定了模型一次能"看到"多少上下文。

## 模块4：数学公式与代码的对应

今天涉及的主要是数据预处理，暂无反向传播的公式。但有一个核心的**矩阵乘法**需要记录：

$$
\text{output} = X \cdot W^T
$$

| 对应代码 | `out = x @ w.T` |
|---|---|
| 对应变量 | $X$: 输入张量 (1, 3)，$W$: 权重张量 (5, 3)，$W^T$: 转置后 (3, 5) |
| 直觉解释 | 一个 1×3 的输入向量经过 5 个神经元（每个神经元有 3 个权重），得到 1×5 的输出 |

**为什么要转置**：线性层的权重习惯存为 `(out_features, in_features)`，即每一行是一个神经元的权重。做矩阵乘法时需要转置成 `(in_features, out_features)` 才能和输入匹配。

## 模块5：今日总结

### 收获

1. **张量 ≠ 列表**：`+`、`*`、`@` 的行为完全不同，这是 PyTorch 的基础认知
2. **广播机制**：从右往左对齐，维度为 1 的自动拉伸——理解了这个，90% 的 shape 报错都能自己排查
3. **滑动窗口构建数据集**：`context = context[1:] + [ix]` 这一行代码浓缩了"给定过去，预测未来"的整个语言模型思想

### 困惑

- `reshape` 和 `view` 的区别到底什么时候重要？目前只知道 `view` 要求内存连续，但什么情况下内存会不连续？
- `block_size=8` 是怎么选的？选大了是不是模型更准？选小了呢？这个超参数有没有经验法则？
- 验证集和测试集各 10% 的比例是不是固定的？数据量少的时候怎么分？

### 术语表更新

| 术语 | 含义 |
|---|---|
| Tensor（张量） | PyTorch 的多维数字数组，支持 GPU 计算和自动求导 |
| Shape（形状） | 张量每个维度的大小，如 `(2, 3, 4)` |
| Broadcasting（广播） | 不同形状的张量自动扩展后进行逐元素运算 |
| `block_size` | 上下文窗口大小，即模型一次看多少个字符 |
| `stoi` / `itos` | String-to-Integer / Integer-to-String 映射字典 |
| `shuffle` | 打乱数据顺序，消除顺序偏差 |
| `view` / `reshape` | 改变张量形状，不改变数据 |
