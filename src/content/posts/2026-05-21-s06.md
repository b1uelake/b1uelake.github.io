# 学习笔记 · s06: Context Compact

今天在 s05 Agent 循环的基础上，搞清楚了如何用三层压缩策略让 Agent 在大项目里无限续航。

---

## 核心概念

### Context Compaction（上下文压缩）

- **直觉/类比**（必填）：把对话历史想象成一张办公桌。桌面空间（context window）有限。Micro-compact 是把三天前的草稿塞进抽屉；auto-compact 是把整张桌子的东西装箱入库、只留一张"本周工作摘要便利贴"；compact 工具是你主动说"现在就打包"。
- **解决什么问题**：没有它，读 30 个文件 + 跑 20 条命令就能把 100k token 窗口塞满，Agent 直接卡死或截断。
- **和已有知识的关系**：是对 s05 Agent 循环的扩展，在每次 LLM 调用前/后插入压缩逻辑，不改变工具数量（仍是 5 个，新增 compact 工具替换占位）。

---

## 结构图

```
Every turn:
+------------------+
| Tool call result |
+------------------+
        |
        v
[Layer 1: micro_compact]        (silent, every turn)
  Replace tool_result > 3 turns old
  with "[Previous: used {tool_name}]"
        |
        v
[Check: tokens > 50000?]
   |               |
   no              yes
   |               |
   v               v
continue    [Layer 2: auto_compact]
              Save transcript to .transcripts/
              LLM summarizes conversation.
              Replace all messages with [summary].
                    |
                    v
            [Layer 3: compact tool]
              Model calls compact explicitly.
              Same summarization as auto_compact.
```

**这张图在说什么**：三层压缩是串联关系，不是并联——每轮必走第一层，超阈值才触发第二层，第三层是随时可手动插入的"应急出口"。越往下，信息损失越大，但换来的上下文空间也越多。

## 代码记录

### Layer 1 · `micro_compact`

**输入**：完整 messages 列表。**输出**：原地修改后的 messages，旧 tool_result 被替换为占位符。

**最关键的一行**：`if len(tool_results) <= KEEP_RECENT: return messages`——只有结果数超过"保留最近 N 条"的阈值才开始压缩，防止把刚产生的结果也丢掉。

```python
def micro_compact(messages: list) -> list:
    tool_results = []
    for i, msg in enumerate(messages):#enumerate表示为对象自动添加计数器
        if msg["role"] == "user" and isinstance(msg.get("content"), list):#content有字符串和列表两种合法形式
            for j, part in enumerate(msg["content"]):
                if isinstance(part, dict) and part.get("type") == "tool_result":
                    tool_results.append((i, j, part))
    if len(tool_results) <= KEEP_RECENT:   # 关键判断：不够多就不压缩
        return messages
    for _, _, part in tool_results[:-KEEP_RECENT]:   # 只压缩"最近N条"之外的旧结果
        if len(part.get("content", "")) > 100:#避免content内容太少反而被压缩多了
            part["content"] = f"[Previous: used {tool_name}]"
    return messages
```

**最重要的现象**：压缩后 token 数量下降，但 Agent 仍知道"曾经用过哪个工具"，只是忘了具体输出内容——代价是可控的。

代码解释：
```python
for i, value in enumerate(['a', 'b', 'c']):
    print(i, value)
# 输出：
# 0 a
# 1 b
# 2 c
```

### Layer 2 · `auto_compact`

**输入**：超过 token 阈值的完整 messages。**输出**：只含一条"压缩摘要"消息的新列表。

**最关键的一行**：`messages[:] = auto_compact(messages)` 在调用处用切片赋值原地替换——这样外部持有的引用不会失效，Agent 主循环不需要重新绑定变量。

**运行结果里最重要的现象**：压缩后 messages 从几十条变成 1 条，但磁盘上的 `.transcripts/transcript_*.jsonl` 保留了完整历史，信息没有真正消失。

```python
def auto_compact(messages: list) -> list:
    # 先把完整对话存盘，丢失前留底
    transcript_path = TRANSCRIPT_DIR / f"transcript_{int(time.time())}.jsonl"
    with open(transcript_path, "w") as f:
        for msg in messages:
            f.write(json.dumps(msg, default=str) + "\n")#msg是messages里的元素，messages是列表，而msg是字典，无法写入到json中，所以dumps的作用就是dump to string（转成字符串）
    # 用 LLM 做摘要，截取前 80000 字符防止摘要本身超限
    response = client.messages.create(
        model=MODEL,
        messages=[{"role": "user", "content":
            "Summarize this conversation for continuity..."
            + json.dumps(messages, default=str)[:80000]}],#+表示字符串的拼接
        max_tokens=2000,
    )
    # 关键：返回只含摘要的单条消息列表，彻底清空旧上下文
    return [
        {"role": "user", "content": f"[Compressed]\n\n{response.content[0].text}"},
    ]
```
LLM 返回的 `response.content` 是一个**列表**，结构大概是这样：
```python
[
    TextBlock(type="text", text="这是摘要内容..."),
    # 可能还有其他block，比如tool_use
]
```
所以response.content[0].text}返回的是text也就是摘要内容。

### Layer 4 · Agent 主循环整合三层

**输入**：初始 messages。**输出**：无限续航的 Agent 对话循环。

**最关键的一行**：`messages[:] = auto_compact(messages)` 出现了两次——一次是阈值自动触发，一次是 `compact` 工具手动触发，共用同一个函数，逻辑不重复。

```python
def agent_loop(messages: list):
    while True:
        micro_compact(messages)                        # 第一层：每轮静默清理旧结果
        if estimate_tokens(messages) > THRESHOLD:
            messages[:] = auto_compact(messages)       # 第二层：超阈值自动摘要
        response = client.messages.create(...)
        # ... tool execution ...
        if manual_compact:
            messages[:] = auto_compact(messages)       # 第三层：工具手动触发摘要，于用户输入/compact不同，用户输入时在agent_loop外，直接运行压缩，无需等下次循环。
```

```python
# 普通赋值：messages 变量指向新列表，原列表不受影响
messages = auto_compact(messages)

# 切片赋值：原地修改列表内容，所有持有该列表引用的地方都会看到变化
messages[:] = auto_compact(messages)

```

## 术语表

|术语|含义|
|---|---|
|`KEEP_RECENT`|micro_compact 保留的最近 N 条 tool_result 数量，超出部分被替换|
|`THRESHOLD`|触发 auto_compact 的 token 数阈值（代码示例中为 50000）|
|`tool_result`|messages 中 role=user 的 content 里，type="tool_result" 的片段，即工具执行返回值|
|transcript|压缩前保存到 `.transcripts/` 目录的完整对话 JSONL 文件，用于灾难恢复|
|`messages[:] = ...`|Python 切片赋值，原地替换列表内容而不改变引用地址|

## 核心问题与解答
<u>为什么必须要compact</u>
读一个 1000 行文件 ≈ 4000 token，读 30 个文件 + 跑 20 条命令就能轻松突破 100k token。上下文窗口是硬性上限，超过之后 API <font color="#ffc000">直接报错</font>，Agent **完全无法继续工作**——不是变差，是直接崩掉。所以压缩不是优化，是续命。


<u>一二层的压缩内容与压缩条件</u>

| 压缩内容 | 触发条件                                               |            |
| ---- | -------------------------------------------------- | ---------- |
| 第一层  | 旧 tool result → 占位符 `[Previous: used {tool_name}]` | 每轮自动运行     |
| 第二层  | 全部 messages保存到磁盘 → LLM 生成的摘要                       | token 超过阈值 |

<u>上下文压缩的缺陷与解决方法</u>

在压缩过程中最容易丢失的是**精确信息**——具体数值、文件路径、完整报错信息。这类信息 LLM 摘要时会自然"模糊化"成大致描述。

**设计启示**：对 Agent 来说，重要的精确信息不能只靠上下文记忆，应该持久化存储——比如你说的写入 JSON 文件，或者结构化日志。
