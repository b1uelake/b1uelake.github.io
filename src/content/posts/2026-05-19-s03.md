## AI Agent 学习笔记 — s03 TodoWrite

今天在 s02 多工具 agent 的基础上，搞清楚了**如何让模型在长任务中不跑偏**：用一个带状态的待办清单 + 追问机制来强制它保持专注。

---

## 真实困惑

nag reminder 是插入到 `tool_result` 里的，但模型看到它会"真的感到被追问"吗？还是说它只是因为上下文里出现了 `<reminder>` 标签就机械地去更新 todo？这两种解释对我理解 prompt engineering 的影响差很大，但我现在还分不清楚。

---

## 核心概念

**TodoManager（待办管理器）**

- **直觉/类比**：像一个项目白板，上面贴着便利贴，每张标"未开始 / 进行中 / 完成"。规则是：进行中的便利贴只能有一张。你得先把当前这张翻到"完成"，才能翻下一张。没有这块白板，团队成员（模型）做完第一件事后就不知道下一步是什么，开始凭感觉行动。
- **解决什么问题**：对话越长，系统提示的影响力越被工具结果稀释。一个 10 步任务，做完 2 步后模型可能已经"忘了"后面还有 8 步要做。
- **和已有知识的关系**：是对 s02 的叠加，不替换原有工具，只是新增了第 5 个工具 `todo` 和一个全局计数器。

---

**Nag Reminder（追问注入）**

- **直觉/类比**：像一个站在旁边的主管，你连续干了 3 件事都没汇报进度，他就过来拍你肩膀问"计划更新了吗？"。不是禁止你继续工作，只是制造一个问责节点。
- **解决什么问题**：模型可能会在前几步更新 todo，之后"忘记"维护它。没有追问机制，待办清单就会变成一次性的摆设。

---

## 结构图

```
+--------+      +-------+      +---------+
|  User  | ---> |  LLM  | ---> | Tools   |
| prompt |      |       |      | + todo  |
+--------+      +---+---+      +----+----+
                    ^                |
                    |   tool_result  |
                    +----------------+
                          |
              +-----------+-----------+
              | TodoManager state     |
              | [ ] task A            |
              | [>] task B  <- doing  |
              | [x] task C            |
              +-----------------------+
                          |
              if rounds_since_todo >= 3:
                inject <reminder> into tool_result
```

**这张图的核心关系**：todo 状态不是存在模型脑子里的，而是存在外部的 TodoManager 里；`<reminder>` 是通过劫持 `tool_result` 注入的，相当于把"计划检查"藏进正常的工具返回流程，模型无法绕过。

## 代码记录

### TodoManager.update()

**输入**：一个 item 列表，每项含 `id`、`text`、`status`（pending / in_progress / done）  
**输出**：渲染好的待办清单字符串（返回给模型看）  
**最关键的一行**：`if in_progress_count > 1: raise ValueError`  
——这一行是整个顺序聚焦机制的物理屏障，不是靠 prompt 说"你只能做一件事"，而是直接在工具层报错，模型无法绕开。

```python
class TodoManager:
    def update(self, items: list) -> str:#items指的是**要更新的 todo 任务列表**
        validated, in_progress_count = [], 0
        for item in items:
            status = item.get("status", "pending")# 取状态，如果没有则默认为pending
            if status == "in_progress":
                in_progress_count += 1
            validated.append({"id": item["id"], "text": item["text"],
                              "status": status})
        if in_progress_count > 1:  # 核心约束：同时只能有一个进行中任务
            raise ValueError("Only one task can be in_progress")
        self.items = validated
        return self.render()#`render()` 的目的是把 `self.items` 这个**列表**转换成**可读的字符串**，然后作为 `tool_result` 返回给 LLM。
```
升华：在类中的单个函数里所有self.xxx=其实都是在给这个函数里的self这个对象定义属性，并且是持久的。与函数中的普通对象不同，普通对象在函数运行完之后自动消失。

### `todo` 工具和其他工具一样加入 dispatch map。

```python
TODO = TodoManager()
TOOL_HANDLERS = {
    # ...base tools...
    "todo": lambda **kw: TODO.update(kw["items"]),# 调用类中的update函数
}
```

### Nag Reminder 注入

**输入**：当前消息列表 `messages`，以及全局计数器 `rounds_since_todo`  
**输出**：在最后一条 `tool_result` 的最前面插入一段文字（原地修改）  
**最关键的一行**：`last["content"].insert(0, {...})`  
——插在 `index=0`，确保提醒出现在所有工具结果之前，不容易被模型忽视。

```python
if rounds_since_todo >= 3 and messages:#`and messages` 的意思是**判断 messages 不为空**，若没有这个代码，输入空信息时下一行代码会报错
    last = messages[-1]
    if last["role"] == "user" and isinstance(last.get("content"), list):#`isinstance(obj, type)` 是 Python 内置函数，用来**判断一个对象是不是某个类型**，返回 `True` 或 `False`。
        last["content"].insert(0, {   # 插在最前面，优先级最高
            "type": "text",
            "text": "<reminder>Update your todos.</reminder>",#有没有逗号都不影响
        })
	# **`<reminder>` 是注入到 `tool_result` 里的提示文本**，让模型在处理工具结果的同时看到这条提醒。用 XML 标签包裹是为了让模型容易识别——这是一个结构化信号
```

## 术语表

|术语|含义|
|---|---|
|`in_progress`|任务状态之一，表示"正在进行"，全局唯一|
|`rounds_since_todo`|计数器，记录自上次调用 `todo` 工具后经过了多少轮|
|nag reminder|追问注入，3 轮未更新 todo 时强制插入提醒文本|
|s03|课程第 3 个 agent harness（框架层），在 s02 基础上增加规划能力|
