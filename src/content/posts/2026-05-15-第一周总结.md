**原题：**  
简述 s01 中 Agent 循环的核心步骤（用文字描述或伪代码均可）。

**参考答案（完整版）：**  
1. 用户 prompt 作为第一条 `user` 消息加入 `messages` 列表。  
2. 将 `messages`、系统提示、工具定义一起发给 LLM。  
3. 将 LLM 的响应作为 `assistant` 消息追加到 `messages` 中。  
4. 检查 `response.stop_reason`：如果不是 `"tool_use"`，则结束循环并返回最终结果。  
5. 如果是 `"tool_use"`，则遍历响应中的所有 `tool_use` 块，执行对应的工具处理函数，收集输出结果。  
6. 将所有工具结果构造成 `tool_result` 类型，以 `user` 角色追加到 `messages` 中。  
7. 跳回第 2 步，继续循环。

**原题：**  
s02 中的 `safe_path` 函数解决了什么问题？它是如何实现的？

**参考答案：**  
- **解决问题**：防止路径逃逸工作区（路径沙箱），避免 agent 访问工作目录外的敏感文件。  
- **实现方式**：  
  1. 将传入的路径 `p` 与工作目录 `WORKDIR` 拼接，再用 `.resolve()` 得到绝对路径。  
  2. 调用 `.is_relative_to(WORKDIR)` 检查该绝对路径是否以 `WORKDIR` 为父目录。  
  3. 如果不是，抛出 `ValueError("Path escapes workspace: {p}")`。  
  4. 如果是，返回该安全路径供后续读写操作使用。

**原题：**  
s03 中引入了“nag reminder”机制。它在什么条件下触发？具体如何注入提醒？

**参考答案：**  
- **触发条件**：当 agent 连续 **3 轮以上没有调用 `todo` 工具**时触发（`rounds_since_todo >= 3`）。  
- **注入方式**：  
  在最后一条 `user` 消息的 `content` 列表的**最前面插入**一个文本块：  
  ```python
  {
      "type": "text",
      "text": "<reminder>Update your todos.</reminder>"
  }
  ```
  这样模型在下一次调用时会首先看到该提醒，被督促更新待办列表。

**原题：**  
s04 的 Subagent 为什么不能（也不应该）拥有 `task` 工具？

**参考答案：**  
- 防止**递归无限生成子代理**。如果 subagent 也能调用 `task`，它可能再派生出新的 subagent，导致无限循环、上下文爆炸或栈溢出。  
- Subagent 的设计目标是**在有限、干净的上下文中完成一个明确的子任务**，并将结果以摘要形式返回给父代理。它不应具备再分解任务的能力。  
- 保持 `task` 工具仅在**顶层父代理**中可用，是一种**递归控制**和**责任边界划分**的设计模式。

**原题：**  
s05 中，一个 Skill 目录的标准结构是怎样的？`SkillLoader` 如何识别一个 Skill？

**参考答案：**  
- **标准结构**：  
  每个 Skill 是一个独立的目录，目录中必须包含一个 `SKILL.md` 文件。该文件的开头有 YAML frontmatter（用 `---` 包裹），其中至少包含 `name` 和 `description` 字段。`---` 之后是 Skill 的详细正文（指令、步骤、示例等）。  
- **识别方式**：  
  `SkillLoader` 会递归扫描指定的 `skills_dir` 目录，找到所有名为 `SKILL.md` 的文件。对于每个文件，解析其 YAML frontmatter，取 `name` 字段的值（若缺失则使用该文件所在目录的名称）作为 Skill 的唯一标识，并存储其描述与正文内容。后续通过 `get_content(name)` 可返回完整的 Skill 正文。

**原题：**  
在 s04 的 Subagent 实现中，`run_subagent` 函数内部有一个 `for _ in range(30)` 的循环限制。为什么需要这个限制？如果没有这个限制可能会发生什么？

**你的答案：**

> 创建的subagent不能超过30个，否则会让系统过载

**评分：错误** ❌

**解析：**

- 这个 `range(30)` 限制的不是 **subagent 的创建个数**，而是 **单个 subagent 内部的最大迭代轮数**（即该 subagent 可以连续调用工具的最大次数）。
    
- 目的是防止一个 subagent 陷入无限循环（例如一直调用工具但不给出最终答案），或因为意外逻辑导致迭代次数过多消耗大量 token 和时间。
    
- 如果没有这个限制，可能出现：
    
    - 死循环（例如模型反复调用同一个工具且总是得到相同结果，永远不退出）。
        
    - token 消耗爆炸，API 费用剧增。
        
    - 程序挂起或超时。
        

**参考答案：**

> 限制单个 subagent 最多执行 30 轮工具调用循环，防止因模型错误或工具反馈异常导致无限循环、资源耗尽或 API 费用失控。若无此限制，可能发生死循环或过度计算。

**原题：**  
s05 的 Skill 加载方式把知识放在 `tool_result` 中而不是系统提示里。这种设计如何影响模型的“注意力”和 token 成本？

**你的答案：**

> 模型不需要识别所有skill的全部内容，大大减少了token成本

**评分：正确，但可更全面** ✅（给出补充）

**解析：**

- 你点出了核心优势：**减少 token 成本**。但可以进一步说明“注意力”方面的影响。
    
- 系统提示中只放 Skill 名称和简短描述（低 token），这样模型在大部分任务中不会被大量无关的详细知识分散注意力。只有当模型明确需要某个 Skill 时，才通过 `load_skill` 获取完整内容（高 token），此时模型会集中注意力处理该 Skill 的详细步骤。
    
- 这也是一种“按需加载”的知识管理模式，兼顾了上下文清洁与知识深度。
    

**补充后的完整参考答案：**

> - **Token 成本**：系统提示中只包含 Skill 名称和短描述（约 100 token/skill），而不是完整指令（可能 2000+ token/skill）。只有模型主动加载某个 Skill 时才会消耗其完整内容对应的 token，显著降低每次请求的基础成本。
>     
> - **注意力影响**：模型不会在处理无关任务时被大量不相关的步骤或例子干扰，可以保持对主要目标的聚焦。加载 Skill 后，模型会在当前上下文中临时获得详细指导，注意力自然集中在这些内容上。整体上提高了有效推理质量。
