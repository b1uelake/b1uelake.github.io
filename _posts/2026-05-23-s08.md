# s08: Background Tasks

今天在 s07 同步阻塞式 Agent 的基础上，搞清楚了如何让慢操作"丢后台"——模型继续干活，任务完成后再注入结果。

---

## 核心概念

**BackgroundManager（后台任务管理器）**

- **直觉/类比**（必填）：像餐厅里的"出票系统"。服务员下单后不会站在厨房等菜做好，而是继续接待其他桌——厨房做完了，铃声响，服务员再去端盘子。这里 LLM 是服务员，子进程是厨房，通知队列是那个铃。
- **解决什么问题**：没有它，`npm install` 跑 3 分钟期间 Agent 只能傻等，无法并行处理用户的其他请求。
- **和已有知识的关系**：是对 s07 阻塞式 bash 工具的扩展——阻塞工具仍然保留，后台任务是额外提供的能力，两者共存。

---

**Notification Queue（通知队列）**

- **直觉/类比**：像微信消息的未读列表。子进程完成时把结果"放进收件箱"，Agent 每次发起新的 LLM 调用前统一"读一遍收件箱"（drain），清空后继续。
- **解决什么问题**：避免子线程直接修改主线程状态（会有竞态条件），所有跨线程通信都经过一把锁保护的队列。

---

## 架构图

```
Main thread                Background thread
+-----------------+        +-----------------+
| agent loop      |        | subprocess runs |
| ...             |        | ...             |
| [LLM call] <---+------- | enqueue(result) |
|  ^drain queue   |        +-----------------+
+-----------------+

Timeline:
Agent --[spawn A]--[spawn B]--[other work]----
             |          |
             v          v
          [A runs]   [B runs]      (parallel)
             |          |
             +-- results injected before next LLM call --+
```

**这张图在表达的核心关系**：主线程（Agent 循环）和子进程之间**不直接通信**——子进程完成后把结果塞进队列，主线程在下一次 LLM 调用前统一取出子进程结果。这样主线程始终单线程运行，并发只发生在 I/O 层（子进程），避免了复杂的并发控制。

## 代码解析

### 1. BackgroundManager 初始化

**输入**：无。**输出**：线程安全的任务管理器对象。

```python
class BackgroundManager:
    def __init__(self):
        self.tasks = {}                  # 任务状态字典，key 是 task_id
        self._notification_queue = []   # 通知队列，子进程完成后写入
        self._lock = threading.Lock()   # 保护队列的互斥锁，防止竞态
```

**最关键的行**：`self._lock = threading.Lock()`——所有跨线程写操作都必须持有这把锁，是线程安全的核心保障。
为什么需要锁：**两个线程同时写同一个变量，结果不可预测。**
以 `_notification_queue` 为例，没有锁时：

```
时间轴：

线程A（子进程完成）：读取队列长度=2，准备在第3位插入...
                                    ↕ 操作系统切换线程
线程B（子进程完成）：读取队列长度=2，准备在第3位插入...
线程B：写入，队列=[item1, item2, itemB]
                                    ↕ 切回线程A
线程A：写入，队列=[item1, item2, itemA]  ← itemB 丢了！
```

这就是**竞态条件（Race Condition）**——谁最后写，谁覆盖别人。

<u>加锁后怎么运行</u>
```python
# 子进程完成时（可能多个线程同时调用）
def _notify_complete(self, task_id, result):
    with self._lock:          # ← 抢锁，抢不到就阻塞在这里等
        self._notification_queue.append(result)
    # with 块结束自动释放锁

# 主线程 drain 时
def drain(self):
    with self._lock:          # ← 同一把锁
        results = self._notification_queue[:]
        self._notification_queue.clear()
    return results
```

```
时间轴：

线程A：抢到锁 🔒 → 写入 → 释放锁 🔓
线程B：          等待... → 抢到锁 🔒 → 写入 → 释放锁 🔓

结果：串行执行，数据完整
```

### 2. 启动后台任务

**输入**：shell 命令字符串。**输出**：立即返回 task_id，子进程异步跑。
run 函数在做什么：把一个耗时命令丢给后台去跑，自己立刻返回。
```python
def run(self, command: str) -> str:
    task_id = str(uuid.uuid4())[:8]          # 生成 8 位短 ID，够用且可读
    self.tasks[task_id] = {"status": "running", "command": command}
    thread = threading.Thread(
        target=self._execute,#_表示python内部方法，`_execute` 本身是阻塞的，target是self跑完命令之后返回的内容。
        args=(task_id, command),#`args` 是传给目标函数的参数元组
        daemon=True               # daemon=True：主进程退出时子线程自动销毁，不会僵尸
    )
    thread.start()
    return f"Background task {task_id} started"  # 立即返回，不阻塞
```
<u>核心过程拆解</u>
`threading.Thread` 内部大致是这样调用 target 的：
```python
self._target(*self._args)
# 即 self._execute(*（task_id, command）)
# 等价于 self._execute(task_id, command)
```

```python
# 1. 造容器，把 _execute 装进去（_execute 没有运行）
thread = threading.Thread(target=self._execute, args=(task_id, command))

# 2. 发车，Thread 内部开辟新线程，在新线程里调用 _execute
thread.start()
#    ↑ 这一句做了两件事：
#    ├─ 开辟一个新线程
#    └─ 在新线程里执行 _execute(task_id, command)

# 3. 主线程继续往下走，不等 _execute 跑完
return f"Background task {task_id} started"
```

**最关键的行**：
`daemon=True` 控制**主程序退出时，这个线程怎么办**。
```
daemon=False（默认）         daemon=True
─────────────────────        ─────────────────────
主程序想退出                  主程序想退出
  ↓                            ↓
等所有线程跑完                 直接退出，线程强制销毁
  ↓                            ↓
才真正退出                    不会有"僵尸线程"残留
```

### 3. 子进程执行与结果入队

**输入**：task_id + 命令。**输出**：结果写入通知队列。
```python
def _execute(self, task_id, command):
    try:
        r = subprocess.run(
            command, shell=True, cwd=WORKDIR,
            capture_output=True, text=True,
            timeout=300                        # 最多等 5 分钟，防止任务永远挂住
        )
        output = (r.stdout + r.stderr).strip()[:50000]  # stdout + stderr 合并，截断防止爆内存
    except subprocess.TimeoutExpired:
        output = "Error: Timeout (300s)"
    with self._lock:                           # 持锁写入，保证队列操作原子性
        self._notification_queue.append({
            "task_id": task_id,
            "result": output[:500]             # 只取前 500 字符注入 LLM 上下文，控制 token 消耗
        })
```
**最重要的现象**：输出被截断两次——先截 50000 字符保留完整记录，再截 500 字符注入 LLM。这是"存储"和"上下文"两个不同的关注点。
##### 1. `try` 是什么意思？
`try` 是"**尝试执行**"的意思，属于 Python 的**异常处理机制**。
```python
try:
    # 尝试执行这里的代码
    r = subprocess.run(...)
except subprocess.TimeoutExpired:
    # 如果出错了，跳到这里处理
    output = "Error: Timeout (300s)"
```
##### 2. `subprocess.run(...)` 的参数分别是什么？

| 参数                    | 含义                                         |
| --------------------- | ------------------------------------------ |
| `command`             | 要执行的 shell 命令，比如 `"ls -la"`                |
| `shell=True`          | 让系统 shell（如 bash）来解析命令，支持管道 `\|`、通配符 `*` 等 |
| `cwd=WORKDIR`         | 设置命令的**工作目录**（在哪个文件夹下执行）                   |
| `capture_output=True` | 捕获命令的输出，否则输出会直接打印到终端而无法读取                  |
| `text=True`           | 让输出以**字符串**形式返回，而非原始字节 `bytes`             |

##### 3. `stdout` 和 `stderr` 是什么？

这是 Linux/Unix 系统的**三个标准流**中的两个：
```
stdin  → 标准输入（程序读取用户输入）
stdout → 标准输出（程序正常的输出结果）
stderr → 标准错误（程序报错时的输出）
```
举个例子，执行 `ls 不存在的目录`：
- `stdout` = 空（没有正常结果）
- `stderr` = `"ls: 不存在的目录: No such file or directory"`
代码里把两者合并 `r.stdout + r.stderr`，是为了**不管成功还是报错，都能捕获到输出**。

##### 4. `with` 表示什么？
`with` 准确的理解是：**"在……期间"** 。

`threading.Lock` 这个类，**提前就写好了**这加锁和解锁这两个方法：
```python
class Lock:
    def __enter__(self):
        self.acquire()   # 加锁
    
    def __exit__(self, ...):
        self.release()   # 解锁（不管里面有没有报错，都执行）
......
# 第一步：创建锁 
self._lock = threading.Lock()

# 第二步：用 with 保护关键代码 
with self._lock: # 进入时：对self自动加锁 
	self._data.append(...)# 退出时：自动解锁（哪怕中间报错也会解锁)
```

### 4. Agent 循环：每轮 LLM 调用前排空队列

**输入**：消息历史列表。**输出**：携带后台结果的 LLM 响应。

```python
def agent_loop(messages: list):
    while True:
        notifs = BG.drain_notifications()    # drain_notifications()做了两件事--取出待处理通知，清空队列。待处理指的是任务已经完成，但agent_loop还未取走的通知。
        if notifs:
            notif_text = "\n".join(
                f"[bg:{n['task_id']}] {n['result']}" for n in notifs
            )
            messages.append({
                "role": "user",              # 以 user 消息形式注入，让 LLM 看到后台结果
                "content": f"<background-results>\n{notif_text}\n</background-results>"
            })
        response = client.messages.create(...)  # 此时 LLM 已知晓所有已完成的后台任务
```

**最关键的行**：`drain_notifications()` 放在 `client.messages.create()` 之前——顺序不能颠倒，否则这一轮的后台结果要等到下一轮 LLM 才能看到。

##### 什么是BG？
它是 **Background（后台）** 的缩写，是一个**后台任务管理器对象**。

```
BG（后台任务管理器）
├── 内部维护一个通知队列
├── 负责派发后台异步任务（比如调用工具、查数据库）
└── 任务完成后把结果塞进通知队列，等待主循环来取
```

## s08 Background Tasks 复习总结

### Q1. 为什么 `_notification_queue` 需要 `threading.Lock()`？

多个子线程可能同时调用 `append()`，列表扩容时需要"分配内存 → 复制数据"多个步骤，中间可能被另一个线程打断，导致数据损坏（竞态条件）。加锁保证同一时刻只有一个线程能写入。

---

### Q2. 守护线程（daemon=True）的作用？

主线程退出时，守护线程会被自动杀掉。若不设 `daemon=True`，主线程结束后后台线程仍在运行，程序无法正常关闭，导致卡住。

---

### Q3. 为什么后台结果用 `"role": "user"` 注入？

`user` 代表外部输入，`assistant` 代表 LLM 自己的输出。后台任务结果是外部环境的反馈，应该用 `user` 注入，LLM 才会把它当作新信息处理。若用 `assistant`，LLM 会误以为那是自己说过的话，导致对话历史被污染。

---

### Q4. 支持长时间任务比改大 timeout 更优雅的做法？

用**心跳检测**替代固定 timeout：把 `subprocess.run()` 换成 `subprocess.Popen`，每隔一段时间用 `process.poll()` 检查进程状态，有新输出就重置计时器，超过一定时间没有任何输出才报错。

```python
process = subprocess.Popen(command, shell=True, stdout=subprocess.PIPE)
last_output_time = time.time()

while process.poll() is None:
    time.sleep(5)
    new_output = process.stdout.read()
    if new_output:
        last_output_time = time.time()
    elif time.time() - last_output_time > 300:
        process.kill()
        return "Error: no progress"
```

### Q5. 单线程主循环的核心权衡和瓶颈？

|         | 好处           | 代价               |
| ------- | ------------ | ---------------- |
| 单线程主循环  | 逻辑简单，无并发 bug | LLM 调用串行，无法并行决策  |
| 通知一次性排空 | 实现简单         | 任务太多时 context 超限 |

**瓶颈场景**：大量后台任务同时完成，通知堆积，一次性注入 LLM 导致 context 超限报错。
