今天在"语言模型只能推理、不能行动"的问题基础上，搞清楚了 Agent Loop 是如何用一个 `while True` 循环让模型真正触碰现实世界的。

---

## 核心概念

### Agent Loop（智能体循环）

**直觉/类比**：你在远程指挥一个工程师干活。你说"帮我列出目录里的 Python 文件"，他列了出来发给你，你看了说"好，现在运行第一个"……这个来回沟通直到任务完成的过程，就是 Agent Loop。**循环存在于你们之间，而不在任何一人脑子里。** 模型是工程师，代码是你，`while True` 是你们之间那条电话线。

**解决什么问题**：单次 API 调用像闭卷考试——模型只能靠记忆推理，看不到文件、跑不了测试。没有循环，每次工具调用都得人类手动把结果粘回去。人类自己就成了那个循环。

**和已有知识的关系**：结构上和训练循环一模一样——"前向计算 → 得到反馈 → 继续迭代"。区别只是反馈从梯度变成了工具执行结果，收敛信号从 `loss < threshold` 变成了 `stop_reason != "tool_use"`。

---

### stop_reason（停止原因）

**直觉/类比**：服务员每次来问"还需要什么吗？"——你说"再来一杯水"（`"tool_use"`），他继续；你说"结账"（`"end_turn"`），循环结束。这个字段是整个 Agent 的唯一心跳开关。

---

### 累积式消息列表（Messages History）

**直觉/类比**：模型每次 API 调用都失忆——它是无状态的。消息列表是我们在外部给它维护的"会议纪要"，每次开会前先把所有纪要发给它，它才能接着上次的进度继续干，而不是每次从零开始。

**和已有知识的关系**：类似 RNN 的隐藏状态 $h_t$——上下文不在模型内部，被显式地从外部传入。

---

## 代码

**输入**：一条用户查询字符串。**输出**：无显式返回值，结果体现在工具的副作用上（文件被创建、命令被执行）。**最关键的一行**：`if response.stop_reason != "tool_use": return`——整个 Agent 的生死就在这一行。

```python
def agent_loop(query):
    messages = [{"role": "user", "content": query}]

    while True:
    # 左边是大模型返回的，右边是输入的
        response = client.messages.create(
            model=MODEL, system=SYSTEM,
            messages=messages, tools=TOOLS, max_tokens=8000,
        )
# client.messages.create指的是向大模型发送请求。`system` — 规则，每次都有，模型始终遵守，`messages` — 对话历史，随循环不断增长。response是请求之后的回复。

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":  # 唯一的退出条件，stop_reason时模型回复中自带的属性。
            return#退出函数

        results = []
        for block in response.content:
            if block.type == "tool_use":
            # run_bash是让电脑执行命令，type，input都是block的某个属性
                output = run_bash(block.input["command"])
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,  # 把结果和对应请求绑定
                    "content": output,
                })

        messages.append({"role": "user", "content": results})
        # 回到 while True 顶部，带着完整历史再问一次模型
```

运行结果里最重要的现象是**循环迭代次数**：让模型创建并验证一个文件，通常迭代 2-3 次——第一次写文件，第二次确认文件存在，第三次报告完成。次数反映了任务的复杂度。

## 学习总结
### 概念理解薄弱点

**`client.messages.create()`**

- 误解为"创建用户信息"
- 实际是向 API 发送请求，等待并接收模型回复

**`response` 对象**

- 不清楚 `stop_reason`、`content` 从哪来
- 实际是 API 自动返回的属性，不需要自己定义

**`system` vs `messages`**

- 不清楚两者区别
- `system` 是每次都带的规则，`messages` 是不断增长的对话历史

**`run_bash`**

- 误以为是让大模型执行命令
- 实际是让**本地电脑**执行，模型只是告诉你跑什么

**`role: "user"` 用于工具结果**

- 疑惑为什么工具结果算 user
- API 只有两个角色，所有发给模型的内容都是 user

---

### 默写时反复出错的点

- `tool_use` 写成 `use_tool`（出现多次）
- `result.append({})` 写成 `result.append[{}]`
- `for block in response.content` 漏掉 `.content`
- `response.stop_reason` 写成 `block.stop_reason`
- `content: output` 写成 `content: results`
- `block.input['command']` 漏掉 `block.`

## 术语表

| 术语             | 含义                             |
| -------------- | ------------------------------ |
| `stop_reason`  | 停止原因，控制循环是否继续的字段               |
| `"tool_use"`   | `stop_reason` 的一个值，表示模型要继续调用工具 |
| `"end_turn"`   | `stop_reason` 的一个值，表示模型认为任务完成  |
| `tool_use_id`  | 工具调用 ID，把某个结果和对应的请求绑定起来        |
| stateless（无状态） | 模型不记得上一次调用，上下文必须从外部传入          |
