## 在 s04（子Agent隔离上下文）的基础上，搞清楚了"按需加载知识"这件事——Skills 系统。

**真实困惑**：系统提示（Layer 1）和 tool_result（Layer 2）本质上都是传给模型的文字，为什么非要分两层？直接把所有 Skill 内容全塞系统提示不行吗？我觉得我能说出"太费 token"。
<u>更深的原因</u>

|维度|全塞系统提示|两层（系统提示 + tool_result）|
|---|---|---|
|Token 成本|每轮固定消耗大量|按需消耗|
|注意力有效性|长提示导致中间内容被忽略|注入时紧邻推理步骤，遵循率高|
|技能冲突|无法动态选择|模型可按顺序/条件加载不同 Skill|
|维护与复用|修改麻烦|目录即技能，即插即用|
|窗口空间|挤占对话历史|只占必要的时候|

## 核心概念
`SKILL.md` 的完整结构大概是这样：

```
---
name: pdf
description: Process PDF files
---

# 正文从这里开始

这里是真正的 Skill 内容，步骤说明，使用方法……
```

`---` 之间的部分 = YAML frontmatter（元信息）  
`---` 之后的部分 = Markdown 正文（实际内容）

---
**Skill 按需加载（Lazy Loading）**

- **直觉/类比**（必填）：就像餐厅的菜单和厨房。菜单（系统提示）只列菜名和简介，客人扫一眼就知道有什么，成本极低。等你真的点了"北京烤鸭"，厨房才把那道菜完整做出来送上来（tool_result 注入完整 Skill 内容）。没有这套机制的话，厨房得把所有菜同时摆在桌上——你还没点菜，座位就已经放不下了。
- **解决什么问题**：没有它，10 个 Skill × 每个 2000 token = 20,000 token 全占着，大部分和当前任务毫无关系。上下文窗口是有限资源。
- **和已有知识的关系**：是 s06（上下文压缩）的"预防版"——压缩是把已经撑大的上下文缩小，Skill 加载是从源头控制，让不需要的内容根本不进来。

---

**两层注入架构（Two-Layer Injection）**

- **直觉/类比**：第一层是"目录"，第二层是"正文"。图书馆的检索系统（目录）让你知道有哪些书，但书本身（正文）只有你走到书架拿起来才真正到手。目录很轻，正文很重，合理的系统把两者分开存放。
- **解决什么问题**：模型需要"感知全局"（知道有哪些能力）和"精准使用"（用到时才加载）之间的平衡。
- **和已有知识的关系**：tool_result 注入是 s02 就建立的机制——Skills 只是把这个槽位用来传"知识"而不是传"命令结果"，本质复用了已有通道。

## 结构图

```
System prompt (Layer 1 -- always present):
+--------------------------------------+
| You are a coding agent.              |
| Skills available:                    |
|   - git: Git workflow helpers        |  ~100 tokens/skill
|   - test: Testing best practices     |
+--------------------------------------+

When model calls load_skill("git"):
+--------------------------------------+
| tool_result (Layer 2 -- on demand):  |
| <skill name="git">                   |
|   Full git workflow instructions...  |  ~2000 tokens
|   Step 1: ...                        |
| </skill>                             |
+--------------------------------------+
```

**这张图在说什么**：模型始终背着一张"轻地图"（Layer 1，知道有什么），只有主动开口要，才会收到"重内容"（Layer 2，具体怎么做）。两层之间的触发点是模型自己调用 `load_skill` 工具这个动作。

## 代码记录

```python
class SkillLoader:
    def __init__(self, skills_dir: Path):
        self.skills = {}
        for f in sorted(skills_dir.rglob("SKILL.md")):
            text = f.read_text()
            meta, body = self._parse_frontmatter(text)
            name = meta.get("name", f.parent.name)
            self.skills[name] = {"meta": meta, "body": body}  # 最关键一行：把 meta（目录）和 body（正文）分开存

    def get_descriptions(self) -> str:  # 给 Layer 1 用：只返回名字+简介
        lines = []
        for name, skill in self.skills.items():
            desc = skill["meta"].get("description", "")
            lines.append(f"  - {name}: {desc}")
        return "\n".join(lines)

# 根据传入的name参数，在skills目录里寻找，如果找到了，则返回这个skill的完整命令--body
    def get_content(self, name: str) -> str:  # 给 Layer 2 用：返回完整正文
        skill = self.skills.get(name)
        if not skill:
            return f"Error: Unknown skill '{name}'."
        return f"<skill name=\"{name}\">\n{skill['body']}\n</skill>"
```

1. **输入**：`skills/` 目录路径；**输出**：`get_descriptions()` 返回轻量摘要字符串，`get_content()` 返回带 XML 包裹的完整内容字符串。
2. **最关键的一行**：`self.skills[name] = {"meta": meta, "body": body}` —— 在初始化时就把"目录"和"正文"物理分离，后续两个方法各取所需，互不干扰。
3. **最重要的现象**：`rglob("SKILL.md")` 递归扫描，意味着 Skill 可以任意嵌套在子目录里，加新 Skill 不需要改任何注册代码，放进去就自动被发现。
4. `rglob` 是 `pathlib.Path` 的一个方法，全称是 **recursive glob**--递归。它的作用是：**在当前目录及其所有子目录、子目录的子目录……（即整个目录树）中，递归地搜索匹配给定模式的文件**。


第一层写入系统提示。第二层不过是 dispatch map 中的又一个工具。
```
用户提问
   │
   ▼
┌───────────────────┐
│ 第一层（系统提示）                           │
│ "Skills available: git: ..., code-review:..."│
│ 始终存在，轻量。                            │
└───────────────────┘
   │
   ▼
LLM 判断需要某个技能 → 调用 load_skill("git")
   │
   ▼
┌──────────────────┐
│ 第二层（tool_result）                       │
│ <skill name="git">                          │
│   完整 git 工作流步骤...                    │
│ </skill>                                    │
│ 按需注入，重。                              │
└──────────────────┘
   │
   ▼
LLM 看到完整指令，开始遵循执行。
```

## 个人错误点：
`get_descriptions()` 出现在系统提示里（第一层），返回所有 Skill 的名字和描述列表。`get_content()` 出现在 tool result 里（第二层），返回某个 Skill 的完整 body。
